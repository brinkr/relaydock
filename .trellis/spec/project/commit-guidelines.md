# Commit Guidelines

## Format

Use a concise subject, usually `type: summary`.

Every commit body must explain:

- Reason: why this change is needed
- Purpose: what the change is meant to accomplish
- Scope: which files or behaviors are included
- Boundary and tradeoffs: what is intentionally excluded or deferred

## Example

```text
docs: confirm native desktop architecture

Reason: previous planning included early WebView candidates and a later correction toward Swift plus Rust. Without an explicit ADR, future implementation could drift back to prototype assumptions.

Purpose: record SwiftUI + AppKit shell + Rust core as the implementation architecture and define Swift/Rust ownership.

Scope: update product docs and architecture decision records.

Boundary and tradeoffs: React prototypes remain visual references only.
```

## Rules

- Do not use empty one-line commits for meaningful project decisions.
- Do not hide architecture changes inside unrelated commits.
- If a change is generated by Trellis, still document why the generated files belong in this repository.
