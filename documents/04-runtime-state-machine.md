# RelayDock 运行态与状态机

更新时间：2026-04-22

## 1. 状态机目标

运行态必须回答这几个问题：

- 现在有没有在工作
- 如果没工作，是断了、停了，还是可恢复
- 当前应该提供什么动作
- 本地端口冲突时怎么处理

## 2. 运行页展示对象

运行与恢复页展示的是：

- `connected`
- `reconnecting`
- `error`
- `recoverable`

不展示纯配置态对象。

## 3. 核心状态

### `connected`

表示映射已建立并处于可用状态。

建议显示：

- 状态
- 延迟
- 运行时长
- 今日失败次数

建议动作：

- 停止
- 打开入口

### `reconnecting`

表示连接断开后正在自动重连。

建议显示：

- 状态
- 当前时长
- 错误摘要

建议动作：

- 立即重试
- 停止

### `error`

表示当前实例失败，未进入稳定连接。

建议显示：

- 错误摘要
- 最近失败次数

建议动作：

- 重连
- 查看日志
- 停止

### `recoverable`

表示该对象属于上次运行集合，但当前没有建立连接。

建议动作：

- 恢复
- 改本地端口
- 清除

说明：

- 这里不应再出现一整套无意义状态列占位
- 也不应该继续显示“停止”这种不贴切动作

## 4. 建议状态流转

典型流转如下：

1. `configured`
2. `starting`
3. `connected`
4. `reconnecting`
5. `error`
6. `recoverable`
7. `cleared`

说明：

- `configured` 主要属于配置态
- `recoverable` 是运行页关心的“恢复对象”
- `cleared` 表示从恢复集合中移除

## 5. 自动重连

自动重连应只作用于：

- 曾经成功运行过
- 因网络原因中断
- 当前仍属于有效恢复集合

不应无限对明显无效配置做重试。

## 6. 保活与断联语义

用户已明确提到 `ServerAliveInterval` 与 `ServerAliveCountMax`。

因此 RelayDock 应支持表达：

- 当前连接是否开启保活
- 保活超时导致的重连
- KeepAlive 相关失败原因

建议在：

- 日志页和诊断页
- 资源登记的连接策略摘要区

体现这类信息。

## 7. 恢复机制

需要支持以下恢复触发源：

- 应用意外关闭后重启
- 用户手动关闭后再次打开
- 电脑休眠后唤醒

恢复机制要区分：

- 自动尝试恢复
- 保留为待恢复但不自动启动
- 用户手动恢复

## 8. 主机级动作

运行页中主机级动作应围绕：

- 恢复全部
- 停止当前运行中
- 清空待恢复

而不是继续把预设启动塞进运行页。

## 9. 页级动作

运行页需要具备页级批量能力，例如：

- 重新检查
- 全部折叠
- 停止全部运行
- 清空恢复列表

说明：

- “停止全部运行”和“清空恢复列表”不应混成一个动作

## 10. 临时本地端口覆盖流程

当本地端口冲突时，运行态需要支持：

1. 发现冲突
2. 显示占用信息
3. 提供自动递增建议
4. 允许用户临时改本地端口
5. 启动实例并记录 override

默认行为建议：

- 本次临时修改不写回配置
