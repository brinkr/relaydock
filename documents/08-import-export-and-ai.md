# RelayDock 导入导出与 AI 友好设计

更新时间：2026-04-22

## 1. SSH 命令导入

SSH 命令导入是资源登记页的重要能力。

原因：

- 用户现实中大量使用多 `-L` 命令
- 这些命令是现有资产
- 工具必须提供低成本迁移路径

## 2. 第一版导入原则

导入时：

- 允许粘贴一整行 SSH 命令
- 解析出每个 `-L`
- 默认把每个端口先拆成独立服务草稿

后续可由用户自行合并为“一个主端口 + 多个附属端口”的服务结构。

## 3. 导入后的自动处理

建议支持：

- 生成服务草稿
- 自动生成唯一别名
- 允许用户手改
- 自动探测服务图标候选
- 自动推断服务类型候选

## 4. 导出与迁移

因为产品不打算：

- 做账号体系
- 做云同步

所以导入导出会非常重要。

第一版建议导出：

- 主机
- 规则
- 预设
- 标签
- provider target 配置
- 别名
- 端口策略

第一版不导出：

- 私钥
- 凭据本体

## 5. 为什么不迁移私钥

即使用户存在 SSH 免登录配置，也不建议第一版尝试迁移整个凭据系统。

更合理的做法是：

- 迁移 RelayDock 自身配置
- 导入后检测本机环境是否具备对应 alias / key / agent
- 缺失时给出提示

## 6. AI 友好要求

用户已明确希望未来能让 AI 参与运维。

因此 RelayDock 应保留结构化接口设计。

至少需要支持：

- 稳定 JSON 导出
- 结构化状态查询
- Host / Rule / RuntimeInstance 查询
- 冲突诊断结果结构化输出
- 批量动作结构化调用

## 7. 适合 AI 调用的能力

示例：

- 查询当前有哪些运行实例
- 查询某条规则为什么失败
- 查询某个本地端口被谁占用
- 为某条规则分配下一个可用本地端口
- 批量启动某个预设

## 8. 设计约束

为了让 AI 友好能力真正可用，后续实现时应尽量避免：

- 只有 UI 文案，没有结构化数据
- 运行状态只存在于渲染层
- 冲突信息只靠弹窗提示，不可程序读取
