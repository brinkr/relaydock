# RelayDock 端口管理基础能力

更新时间：2026-04-22

## 1. 定位

端口管理不是 RelayDock 的独立一级产品，而是其共享底层能力。

它同时服务：

- 隧道启动前的端口分配
- 运行中的冲突检测
- 本机开发场景中的临时端口排障

## 2. 为什么必须做

用户已经明确遇到真实问题：

- 本地某个端口先被其他服务占用
- 新项目自动递增到下一个端口
- 调试器/调用方仍连旧端口
- 用户并不知道旧端口是谁占用的
- 过去只能手敲命令排查和处理

RelayDock 本来就需要处理端口冲突，因此应顺带把这类问题解决掉。

## 3. 第一版能力范围

必须支持：

- 扫描本机端口占用
- 识别哪个进程在占用端口
- 冲突提示
- 自动递增分配端口
- 手动指定可用端口
- 释放占用进程

第一版不做：

- 本地项目生命周期管理
- 长期登记本地项目
- 代替 npm / Metro / Docker Compose

## 4. 自动分配策略

建议规则：

1. 优先尝试用户配置端口
2. 如果被占用，提示冲突
3. 提供自动递增结果
4. 用户可手动覆盖
5. 运行态记录本次实际端口

## 5. 典型冲突流程

以 `8088` 为例：

1. 用户预期服务占用 `8088`
2. 系统检测到 `8088` 已被占用
3. 显示：
   - 占用进程名
   - PID
   - 命令行摘要
4. 提供选项：
   - 改用 `8089`
   - 结束占用进程
   - 手动输入其他端口
5. 如果用户选择 `8089`，本次运行建立 `LocalPortOverride`

## 6. UI 落点

这套能力适合落在：

- 启动前冲突弹窗
- 行内“改本地端口”动作
- 日志与诊断里的端口诊断面板
- 全局端口扫描工具位

而不是单独做很重的“本地项目管理页”。

## 7. 本地端口表达

建议表达规则：

- 同端口 loopback：显示单端口
- 端口不同：显示 `本地 -> 远端`
- 远端非 loopback：显示 `本地 -> 远端IP:端口`

## 8. 释放进程

需要明确两个原则：

- 这是辅助动作，不是产品主轴
- 必须明确显示将要结束哪个进程

建议仅对：

- 本地用户可结束的进程
- 且具备足够识别信息

提供该动作。
