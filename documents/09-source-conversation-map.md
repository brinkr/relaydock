## 2. 核心来源

### 会话 A

- 文件：
  - `/Users/brink/.codex/sessions/2026/03/31/rollout-2026-03-31T10-15-28-019d41ac-abaa-7e23-bad3-676c3a4d0a88.jsonl`
- 作用：
  - 这是产品定义与 UI/交互主线

主要沉淀：

- 从“本地服务入口”演进到“隧道与服务工作台”
- 明确不是浏览器、不是后台
- 明确资源登记 / 运行与恢复 / 日志与诊断 / 设置这套结构
- 明确 Host 优先、运行态与配置态分离
- 明确默认浅色、桌面原生感、去卡片/去 Dashboard
- 明确服务行、主机分组、状态动作联动
- 明确运行页不应该继续承担预设启动中心角色

其中高密度定规格的部分主要集中在：

- 用户快速拍板主机优先、列表结构、字段布局的连续问答段
- 关于“当前转发”应改成“运行与恢复”的讨论段
- 关于 `via/provider`、状态列、待恢复、批量动作的多轮修正段

### 会话 B

- 文件：
  - `/Users/brink/.codex/sessions/2026/03/23/rollout-2026-03-23T09-40-08-019d1859-731d-78a0-9f66-583c4ce6415b.jsonl`
- 作用：
  - 这是网络场景与 provider 模型主线

主要沉淀：

- 公司网络与家庭网络差异
- Tailscale 在家庭场景下质量明显更好
- 公司网络下可能需要广州机器作为跳板
- 工具不需要关心 provider 的内部实现细节
- 对用户而言，“连接到哪个目标”才是产品应该表达的内容
- Host 默认 / Rule 覆盖 / Preset 派生 这条路线是合理的

  - `/Users/workspace/relaydock`

  - `https://github.com/brinkr/LocalPort`
  - `/Users/workspace/LocalPort`

- 将 LocalPort 明确为 Gemini/React 静态原型参考，只吸收桌面壳层、信息密度和交互语义，不吸收技术栈和组件结构
