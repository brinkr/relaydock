# RelayDock 产品基线

更新时间：2026-04-22

## 1. 项目身份

- 产品名：`RelayDock`
- 源码目录：`/Users/workspace/relaydock`
- Git 仓库名：`relaydock`
- 默认分支：`main`

## 2. 一句话定义

RelayDock 是一个本地优先的桌面隧道与端口转发工作台，专注于把远端/内网服务稳定地接回本地，并提供运行恢复、冲突处理和批量操作能力。

## 3. 产品不是什么

RelayDock 不是：

- 浏览器
- SSH 终端客户端
- 会话管理器
- 远程工作台
- 运维后台
- 本地开发环境总控平台

它不负责：

- 终端会话管理
- 标签页 / 分屏 shell
- 文件管理 / SFTP
- 远程桌面聚合
- 云账号体系
- 云同步

## 4. 核心定位

RelayDock 只专注于一件事：

- 隧道管理
- 端口转发
- 运行恢复
- 冲突诊断

它的中心价值是：

- 管理越来越多的远程服务到本地的映射
- 让用户不再手动维护大量 `ssh -L` / provider 连接
- 在家庭、公司、跳板等不同网络场景下，切换合适的连接目标
- 在本地端口冲突时快速诊断和处理

## 5. 真实痛点

### 5.1 远程服务越来越多

用户手上有越来越多的：

- 云主机
- 远程容器
- 内网主机
- 开发环境

这些服务往往不能直接暴露到公网，因此必须通过 SSH、Tailscale 等方式转发回本地使用。

### 5.2 手动管理已经不可持续

一开始用户可以手动维护：

- `ssh -L`
- 本地端口
- 浏览器地址
- 调试脚本

但随着服务数量和主机数量增长，手动方式会迅速失控。

### 5.3 全都落在本地端口上

即使最终服务都访问本地地址，仍然会遇到：

- 本地端口冲突
- 自动递增后调用方仍连旧端口
- 浏览器入口分散
- 难以知道某个端口到底被谁占用

## 6. 用户场景

### 6.1 远程服务映射到本地

例如：

- `SSH -> 本地端口`
- `Tailscale -> 本地端口`
- `广州跳板 SSH -> 广州 Tailscale -> 春川服务`

### 6.2 同一主机下存在很多服务

用户可能用一条多 `-L` 命令图省事，但工具里仍应允许按独立服务登记和管理。

### 6.3 服务不一定是 Web

服务可能是：

- Web 管理台
- API
- 数据库
- 远程桌面
- ADB / redroid
- 调试端口
- 其它 TCP 服务

因此产品模型不能只面向 Web。

### 6.4 网络场景切换

已确认存在：

- 家里：Tailscale 直连质量通常更好
- 公司：Tailscale/WireGuard 特征可能受限
- 公司：需要经广州机器再中转到目标主机

因此工具需要表达“当前规则应该连哪个目标”，而不是把底层协议写死。

## 7. 范围边界

## 7.1 重点功能

RelayDock 的重点功能是：

- 主机管理
- 规则/服务管理
- Provider target 管理
- 预设管理
- 运行中实例管理
- 恢复集合管理
- 冲突处理

## 7.2 非重点但可覆盖的基础能力

本地端口管理属于基础能力，不是一级产品模块。

当前只计划覆盖：

- 扫描本机端口占用
- 冲突提示
- 自动分配可用端口
- 释放占用进程

当前不做：

- 本地项目生命周期管理
- 本地项目面板
- 本地项目统一登记
- 一键启动 npm / Docker Compose / Metro / Vite 等

原因是：

- 这不是产品重点
- 但端口诊断本来就是隧道管理所必须具备的底层能力
- 顺手覆盖本地端口冲突，用户收益很大

## 8. 已确认的功能原则

### 8.1 配置态和运行态分离

配置态负责定义：

- 主机
- 规则
- 预设
- 连接目标

运行态负责表达：

- 运行中
- 重连中
- 异常
- 待恢复

### 8.2 主机优先

运行态列表按主机分组展示，支持折叠。

### 8.3 只显示相关主机

运行与恢复页只显示当前有：

- 运行项
- 异常项
- 恢复项

的主机。

### 8.4 状态与动作必须联动

不同状态下应呈现不同动作，而不是所有行都给一套相同按钮。

例如：

- 运行中：停止
- 异常：重连 / 查看日志 / 停止
- 重连中：立即重试 / 停止
- 待恢复：恢复 / 改本地端口 / 清除

### 8.5 端口冲突必须有明确处理策略

对于本地端口冲突，应支持：

- 识别占用方
- 提示冲突
- 自动递增分配新端口
- 手动指定新端口
- 结束占用进程

## 9. 数据模型方向

当前确认的领域对象包括：

### Host

表示物理机、云主机、内网主机或逻辑连接目标。

建议字段：

- 名称
- IP/域名
- 连接身份
- OS / 发行版识别结果
- 标签 / 分组
- 可用 provider targets

### Rule / Service

表示一个可访问的远端能力。

建议字段：

- 服务名
- 本地别名
- 目标主机
- provider target 引用
- 远端目标地址
- 主端口
- 附属端口
- 图标推断信息
- 类型推断信息
- 标签

说明：

- 不强迫所有多端口都建成 TunnelGroup
- 对当前产品来说，`Host + Rule` 比 `TunnelGroup` 更符合用户心智

### Preset

表示一组规则的配置级集合，用于批量启用。

当前方向：

- Preset 属于配置层，不属于运行页
- 后续应支持基础预设 + 覆盖/派生

### ProviderTarget

表示某条规则实际要连接的目标。

第一版至少支持：

- `ssh`
- `tailscale`

说明：

- 工具关心“当前规则连哪个目标”
- 不负责实现 provider 自身组网逻辑

### RuntimeInstance

表示某条规则当前的运行实例。

### RecoveryItem

表示：

- 上次运行过
- 当前未连接
- 但用户可能希望恢复

的对象。

### LocalPortOverride

表示一次运行中对本地端口的临时改写。

这个 override：

- 不一定写回规则本体
- 但必须被运行态感知

### PortUsage / PortClaim

表示本机端口被谁占用、能否释放、是否冲突。

这层能力供两类场景复用：

- 隧道规则启动前分配端口
- 本机端口冲突诊断

## 10. 页面职责

### 资源登记

这是配置中心。

负责：

- 维护主机
- 维护规则
- 维护预设
- 维护 provider targets
- 导入 SSH 命令
- 从多 `-L` 语句中拆规则草稿
- 自动探测图标/别名候选

已确认的交互方向：

- 左侧主机列表，右侧当前主机详情
- 支持手动分组/标签过滤
- `新主机` 是页面最明确的 Primary CTA
- 新增服务/规则优先用弹窗

### 运行与恢复

这是运行态工作区。

只负责：

- 当前运行中的映射
- 重连中的映射
- 异常项
- 待恢复项

不负责：

- 定义预设
- 维护规则
- 展示所有在线主机

### 日志与诊断

负责：

- 运行日志
- 重连原因
- 失败原因
- 健康检查结果
- 冲突诊断

### 偏好设置

保持极简收敛，不做后台式复杂设置结构。

## 11. 视觉与交互基线

### 11.1 默认中文

软件默认中文，但不强迫把所有常见开发英文词都翻译掉。

### 11.2 默认浅色

可以支持深色，但默认不是黑暗模式。

### 11.3 桌面原生感优先

已确认的壳层方向：

- 红黄绿交通灯
- 可拖拽标题栏
- source list 风格侧边栏
- 原生工具栏
- 底部状态栏

### 11.4 不是 Dashboard，不是网页后台

明确否定：

- KPI 卡片首页
- 大白卡套大白卡
- SaaS / Admin 风格
- 大量悬浮卡片和过强投影

### 11.5 高密度、列表优先

目标气质应更接近：

- IDE
- 专业桌面工具
- Surge 一类控制台工具

而不是营销页或普通后台。

## 12. 图标与识别基线

### 主机图标

需要识别：

- OS
- 发行版

并在列表中展示。

### 服务图标

不应只依赖 favicon。

还应综合推断：

- 服务名
- 常见端口
- 协议类型
- 项目图标

例如：

- RDP
- SQL
- Redis
- RabbitMQ
- Grafana
- Jupyter

### Provider 表达

Provider 更适合用：

- 图标
- 短标签

而不是很重的长文案。

## 13. 导入导出与迁移

当前方向：

- 不做云同步
- 不要求用户登录

但未来应支持：

- 配置导出
- 配置导入
- 跨机器迁移

第一版建议导出：

- 主机
- 规则
- 预设
- 标签
- provider target 配置
- 本地别名
- 端口策略

第一版不导出：

- 私钥
- SSH 凭据本体

导入时应尽量做：

- 环境校验
- 缺失 alias 提示
- provider target 不可用提示

## 14. AI 友好要求

RelayDock 应从一开始就保留对 AI / 自动化友好的设计。

建议预留：

- JSON 导入导出
- 结构化状态查询
- 按主机 / 规则 / 实例查询
- 冲突诊断结果可机器读取
- 批量动作可脚本化

## 15. 命名与项目约定

- 产品名：`RelayDock`
- 仓库名：`relaydock`
- 本地源码目录：`relaydock`
- GitHub Pages 路径：`/relaydock/`

命名约定：

- 给用户看的品牌名：`RelayDock`
- 给路径和机器看的项目名：`relaydock`

## 16. 当前明确不做的事情

- 终端与会话管理
- SFTP / 文件管理
- 本地项目生命周期管理
- 云账号体系
- 云同步
- 复杂商业化能力

## 17. 下一阶段建议

正式开发前，建议按以下顺序推进：

1. 固化正式领域模型
2. 固化状态机
3. 确定桌面技术栈
4. 先做本地配置存储与运行态存储
5. 先接 SSH provider
6. 再接 Tailscale provider
7. 再补导入导出、图标探测、日志增强
