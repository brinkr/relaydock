# RelayDock 范围与非目标

更新时间：2026-04-22

## 1. 核心问题

RelayDock 解决的是：

- 大量远程服务接入本地后的管理失控
- 本地端口转发越来越多以后，难以知道：
  - 哪些服务正在运行
  - 哪些服务断了
  - 哪些服务可以恢复
  - 当前本地端口被谁占用
  - 应该走哪个 provider target

## 2. 一级目标

RelayDock 的一级目标只有一个：

做一款专注于隧道管理和端口转发的桌面工具。

它的一级功能包括：

- 主机管理
- 规则/服务管理
- Provider target 管理
- 预设管理
- 运行中实例管理
- 恢复集合管理
- 冲突诊断
- 批量启动/停止/重连/恢复

## 3. 非目标

RelayDock 明确不做以下方向：

- SSH 终端
- 会话标签/分屏
- 远程 shell 工作台
- SFTP / 文件浏览器
- 远程桌面管理器
- 运维平台
- 云同步产品
- 登录态产品
- 协作型 SaaS

这些能力即使将来可能存在集成需求，也不属于产品主轴。

## 4. 为什么不做终端与会话管理

用户已经明确拍板：

- 当隧道是通的以后，用户会自行在终端或其它工具中利用已有隧道工作
- RelayDock 不承担“终端本体”的角色

这样做的好处：

- 产品边界清晰
- 实现难度明显下降
- 不会滑向 Electerm/Termius/Termix 这类全家桶路线
- 所有交互都围绕“映射是否工作”和“如何恢复/切换”展开

## 5. 本地端口管理的边界

本地端口管理不是一级模块，而是基础能力。

它存在的原因不是要做“本地开发环境总控台”，而是：

- 隧道管理本来就离不开本地端口占用判断
- 用户日常开发时的本地端口冲突问题，与这套底层能力高度重合

第一阶段只覆盖：

- 扫描本机端口占用
- 冲突提示
- 自动分配可用端口
- 结束占用进程

第一阶段明确不做：

- 本地项目生命周期管理
- 本地项目配置登记
- 本地项目长期资源化
- 一键启动 npm / Metro / Docker Compose / Vite

## 6. 用户价值边界

RelayDock 的主价值来自两层：

### 第一层：远程服务接入本地

这是产品的核心价值。

### 第二层：统一处理端口冲突

这是产品的基础设施价值。

第二层必须服务第一层，但不能反过来把产品拖成“本地端口工具”。

## 7. 成功标准

如果 RelayDock 第一阶段做对了，用户应该能做到：

- 在一个桌面工具里看到当前有哪些映射正在工作
- 在应用重启或机器休眠后恢复上次运行集合
- 在家庭/公司/跳板不同场景下切换到正确的连接目标
- 在本地端口冲突时，不再手敲命令去排查和处理
